# Contributing

Thanks for helping improve `Review`.

## Good Contributions

- sharpen review heuristics without making them brittle
- improve readiness, requirements, or risk coverage
- add examples that make the skill easier to use
- refine templates so outputs are easier to act on
- improve wording, structure, and documentation

## Before You Open a PR

- keep review requests read-only by default
- preserve the evidence-first approach
- avoid overfitting the skill to one framework or one repo shape
- prefer concise instructions over large prompt blocks
- make sure example prompts still match real usage

## Pull Request Notes

Please include:

- what changed
- why it improves the skill
- any tradeoffs or behavior changes
- example prompts or before/after output when relevant

## Style Expectations

- use clear, direct language
- keep sections skimmable
- avoid redundant rules
- prefer reusable guidance over one-off edge cases

## Validation

If you have a local Codex skill validator available, run it before opening a pull request. At minimum, make sure:

- the skill structure is intact
- links and relative paths are valid
- the examples still match the documented behavior
